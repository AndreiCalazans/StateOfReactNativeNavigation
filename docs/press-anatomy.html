Anatomy of a press:
from touch to paint

A press in React Native crosses four threads and three layer boundaries before a new pixel appears on screen. This post traces every hop — with real Systrace and Hermes CPU-profile data — and answers: where can you accidentally block responsiveness, and how do you instrument it in production?

New Architecture (Fabric + JSI) Android · Samsung Galaxy A16 data from live Perfetto traces + Hermes profiles

The data below comes from the navigation performance repo. A tap on open-details was recorded inside a Perfetto system trace (atrace + scheduler) while a Hermes sampling profile ran in parallel. Everything is measured from the real device.

Thread legend: UI = main Android UI thread (also called com.rnperf.xxx in the trace). JS = mqt_v_js, the JS engine thread. native = mqt_v_native helper. RT = HWUI RenderThread. SF = SurfaceFlinger (separate process, not shown here).

What the Hermes CPU profile shows inside JS

During the press, the JS thread is doing this work (from the source-mapped Hermes profile of the React Navigation app's navigate-to-Details capture). Self-time = time the function was the actual leaf executing, not including callees.

── JS thread, press-handling burst (~80ms wall, ~80ms JS busy) ──

16.4ms  [HostFunction] completeRoot          ← Fabric commit: synchronous JSI call into C++
11.9ms  beginEvent                           ← synthetic event dispatch machinery
10.9ms  updateCallback                       ← React event handler invocation
10.7ms  [HostFunction] appendChild           ← per-view JSI call (x calls for new screen)
  7.5ms  [Native] functionPrototypeApply      ← JS engine intrinsic (spread/apply calls)
  6.7ms  destructor                           ← synthetic event pool cleanup
  5.9ms  releasePooledEvent                   ← synthetic event pool
  5.7ms  [Native] mapPrototypeGet             ← Map.get (navigation state lookups)
  4.2ms  emit                                 ← EventEmitter dispatch (navigation events)

Blame by library:
 34.6ms  Native                               ← JSI/C++ calls (completeRoot + appendChild)
 28.6ms  react-native                         ← React reconciler + event system
 10.9ms  @react-navigation                    ← navigation state updates
  5.7ms  @expo                                ← expo-modules-core routing
Key insight: [HostFunction] completeRoot is the dominant JS-side leaf. It's a synchronous JSI call from JS into Fabric's C++ layer — JS is fully blocked for its entire duration. On a trivial screen it's ~12 ms; on the 24-row heavy screen it grew to ~60 ms. Everything inside Fabric (layout via Yoga, shadow tree diff, mutation-list serialization) happens inside that call, invisible to the Hermes profiler.

Where you can accidentally block responsiveness

Every stage of the pipeline is an opportunity to add latency. Here are the real ones:

🔴 Long synchronous work inside onPress
JS thread, step ②
Your onPress handler runs synchronously on the JS thread before any reconciliation. Expensive computation, large JSON.parse, synchronous storage reads, or chained .forEach loops on large arrays here directly delay the commit. Rule: keep onPress handlers under 5 ms. Move heavy work to setTimeout(fn, 0) or a Worker.
🔴 Large re-render subtree from a state update
JS thread, step ② — workLoopSync / beginWork
React reconciles the entire subtree rooted at the component whose state changed. If your onPress calls setState high in the tree (e.g., a context that wraps the whole app), thousands of components may run render. Use React.memo, split context, or move state down.
🔴 Heavy destination screen (large Fabric commit)
JS thread, step ② — [HostFunction] completeRoot
Measured: trivial screen = ~12 ms completeRoot; 24-row heavy screen = ~60 ms. Every node is a synchronous JSI call into C++. Virtualize lists, lazy-render off-screen content, and avoid mounting large trees synchronously on navigation. Consider startTransition / deferred rendering for expensive screens.
🟡 Animation running while JS is reconciling
UI thread (Choreographer) + JS thread contention
If a CSS-style animation or Reanimated worklet needs to run on the JS thread while your onPress reconciliation is happening, one of them will skip frames. Use Reanimated's worklet thread (useAnimatedStyle with worklet functions) to keep animations off the JS thread entirely.
🟡 Synchronous native module calls in onPress
JS thread, step ② — [HostFunction] getConstants
Native modules that call getConstants() synchronously (the Expo module proxy does this for every module's first access) block the JS thread. Cache constants at module load time — they're already loaded during cold start.
🟡 Event pooling churn (legacy event system)
JS thread — destructor, releasePooledEvent
The synthetic-event pool still churns in react-navigation builds (visible in the profile as beginEvent + releasePooledEvent ≈ 12 ms). This is framework overhead but is worth knowing: it means the JS thread is busy for a full frame after the press before the reconcile even starts.
🟢 EarlyPostImeInputStage delay (Android / OEM)
UI thread — before React sees the event
The 9 ms EarlyPostImeInputStage (two binder calls to InputManagerService) is an Android / OEM tax — seen on Samsung One UI. There is nothing React Native can do about it. It's real latency for the user but not attributable to your app code. On stock Android / Pixel it is typically <2 ms.

How to instrument this in production

Level 1 — JS-only instrumentation (zero native code)

Wrap every onPress handler to measure JS execution time. Pair performance.now() at press entry with a requestAnimationFrame callback for when the next frame is scheduled:

// utils/pressTracing.ts
import { InteractionManager } from 'react-native';

export function tracedPress(
  label: string,
  handler: () => void,
  onMeasure?: (result: PressTimings) => void,
) {
  return () => {
    // ① time the JS handler + reconcile
    const t0 = performance.now();
    handler();
    const jsMs = performance.now() - t0;   // everything up to completeRoot

    // ② raf fires AFTER mount items are applied (next Choreographer frame)
    const t1 = performance.now();
    requestAnimationFrame(() => {
      const rafMs = performance.now() - t1; // ≈ mount + vsync wait
      const totalMs = performance.now() - t0;
      onMeasure?.({ label, jsMs, rafMs, totalMs });
    });
  };
}

// ③ When the page is done with heavy async work:
InteractionManager.runAfterInteractions(() => {
  // fires after animations + interactions complete
  reportMetric('interaction_settled');
});

Attach it to any Pressable:

<Pressable
  onPress={tracedPress('open-details', () => router.push('/details'), result => {
    analytics.track('press_timing', result);
  })}
/>

Level 2 — React Profiler (dev / staging builds)

The React Profiler API wraps a subtree and gives you render/commit timing with no native instrumentation required:

import { Profiler } from 'react';

function onRender(
  id: string,
  phase: 'mount' | 'update',
  actualDuration: number,   // time rendering committed subtree (ms)
  baseDuration: number,     // estimated cost without memoization
  startTime: number,
  commitTime: number,
) {
  if (actualDuration > 16) {
    console.warn(`[Profiler] ${id} ${phase} took ${actualDuration.toFixed(1)}ms`);
  }
}

<Profiler id="DetailsScreen" onRender={onRender}>
  <DetailsScreen />
</Profiler>

Note: Profiler has non-trivial overhead in DEV mode; disable in production unless the build is compiled with profiling enabled (react-native --profile). actualDuration covers JS reconcile but not the native mount (MountItemDispatcher) or the RenderThread frame.

Level 3 — atrace markers from JS (release builds)

The @react-native-community/systrace package (or the raw Systrace API from react-native) lets you emit custom slices visible in Perfetto:

import { Systrace } from 'react-native';

onPress={() => {
  Systrace.beginEvent('MyApp.openDetails');
  navigation.navigate('Details');
  Systrace.endEvent();
  // completeRoot fires *before* the endEvent in JS, so the
  // Fabric commit is inside the slice. Visible in ui.perfetto.dev.
}}

To see this in Perfetto, your release build must be profileable (<profileable android:shell="true"/>) and you run a trace with atrace_apps: "*" — exactly what the rn-perf-tooling config plugin and perfetto-trace.sh script in this repo do.

Level 4 — Flashlight continuous monitoring

For production monitoring of smoothness across the full press interaction (not just the first frame), Flashlight's measure command (run via Maestro) gives you FPS and CPU across the entire flow. It won't break down the JS hop boundaries but gives aggregate responsiveness signal that's cheap enough to run in CI:

# from perf-tooling:
perf-tooling/scripts/flashlight-measure.sh \
  --app com.rnperf.reactnavigation \
  --flow perf-tooling/maestro/navigate.yaml \
  --label react-navigation

Level 5 — Hermes sampling profiler in release

For a deep JS call-stack trace in a production build, use react-native-release-profiler (already set up in every app in this repo via the rn-perf-tooling plugin):

// In your JS entry before anything else:
import { startProfiling, stopProfiling } from 'react-native-release-profiler';

startProfiling();               // begins Hermes sampling

// … after the interaction you want to profile:
const path = await stopProfiling(true); // dumps to /sdcard/Download/

// pull + source-map on your machine:
// node perf-tooling/scripts/convert-hermes-profile.js \
//   --in raw.cpuprofile.txt --out profile.json --sourcemap index.map

Open the converted JSON in Chrome DevTools → Performance → load profile for a full flame chart of every JS function called during the press.

Summary — the press round-trip in numbers

PhaseThreadTypical costYour code's contribution
Input dispatch (Android)UI9–14 msNone — OS + OEM code
React hit-test (TouchTargetHelper)UI1–4 msProportional to shadow-tree depth
JS event routing (topTouchStart/End)JS1–12 msLibrary overhead (synthetic events)
Your onPress handlerJS0–∞100% your code
React reconciliationJS1–30 msProportional to re-render subtree size
Fabric commit (completeRoot)JS5–60 msProportional to new node count
Native mount (MountItemDispatcher)UI2–4 msProportional to mutation list size
HWUI draw + eglSwapRT3–8 msProportional to dirty layer count
The rule of thumb: for a press to feel instant, the sum of all JS-thread phases must complete within one vsync (~16.7 ms at 60 Hz, ~11 ms at 90 Hz). The OS input delay (~9–14 ms) is already eating most of that budget before your code runs. Every millisecond you spend in onPress, reconciliation, or completeRoot is a millisecond the user waits for a response.
